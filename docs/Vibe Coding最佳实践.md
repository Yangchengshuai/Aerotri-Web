

您总结的工程级方法论非常全面且实用，涵盖了从项目规划到实施再到维护的各个方面。下面是您的经验总结，包括序号：

1. **Plan 模式先行**：复杂任务先制定 Plan，再 Coding。Plan 是“活文档”，可随开发迭代更新。
2. **多角色共创 PRD**：模拟高级前端、后端、算法专家、产品经理，多轮讨论澄清需求，形成可执行 PRD 和开发 Plan。
3. **角色拆分驱动实现**
    - 前端关注 UI/交互
    - 后端聚焦数据和系统稳定
    - 产品经理关注使用流程
    - 算法工程师关注假设与边界
4. **测试工程师视角**：每个功能集成后先测试，失败立即修改 Plan，基于真实测试数据验证功能可用性。
5. **方案调研先行并文档化**：使用 ChatGPT 或深度研究调研方案，沉淀为 .md 文档，再输入 AI 优化整体 Plan。
6. **明确验收标准**：文档阶段明确“完成标准”，避免功能无限扩展或返工。
7. **提供官方文档和示例**：避免 AI 使用错误接口或加载错误。
8. **少量真实数据驱动迭代**：少量真实数据比大量描述更有效，可持续测试、快速迭代。
9. **优先使用高级模型**：高级模型理解复杂需求和工具能力更强，可减少反复修改成本。
10. **先优化 Prompt，再交给 AI 执行**：好的 Prompt 是高效执行的前提。
11. **引入 MCP / Skills**
    - Playwright MCP 自动操作浏览器
    - 前端 Skills 去除“AI味”
    - Git Skills 规范工作流
12. **文档驱动开发（DDD）持续跟踪**：文档持续更新，用于跟踪开发、测试结果和经验沉淀。
13. **Git + AI 协作可工程化为 Skill**：分支策略、提交规范、代码检查流程可封装为 AI Skill 提高一致性。
14. **AI 理解确认与测试数据**：每个关键功能提供测试数据，多轮对话确保 AI 理解需求。
15. **技术路线文档与环境信息全量提供**：CUDA、依赖库、算法框架版本一次性给全，必要时使用 conda / env 隔离。
16. **AI 死循环及时人工介入**：多次尝试仍无进展，问题在方向，而非尝试次数。
17. **低成本解决环境问题**：重启 Docker / Server 有时能解决缓存或状态异常问题。
18. **前端 F12 控制台报错必备**：高效定位前端 Bug。
19. **算法链路先手动跑通再集成**：单模块验证正确后再做系统集成，避免问题叠加。
20. **提供参考用例进行对比验证**：目录结构、输入输出结果对比，快速判断实现偏离。  
21. 增加每个关键阶段的运行log的输出和文件保存，输给AI进行问题快速定位。
